「まずAPIを作って、それからフロントを繋げましょう」

会議でエンジニアがそう言った。周りは頷いている。自分だけ「？」が浮かんでいる。

「APIって...何？」

聞きたい。でも、今さら聞けない。みんな当然のように話しているし。

帰ってググった。「Application Programming Interface」。ふーん、で、何？
Wikipedia を読んだ。「ソフトウェアコンポーネント同士が...」。余計わからない。
ChatGPT に聞いた。「APIとは、異なるソフトウェア間でデータや機能をやり取りするための...」。いや、だから何なの。

この記事は、そんなあなたのための記事です。

技術的な定義はいったん忘れてください。レストランに行ったことがあれば、APIは理解できます。

## APIとは「注文の仕組み」である

レストランで考えてみましょう。

あなたはお客です。お腹が空いている。カルボナーラが食べたい。

でも、厨房に入って自分で作ることはできません。代わりに、メニューを見て「カルボナーラください」と注文します。しばらくすると、料理が運ばれてきます。

この「注文の仕組み」が API です。

```
┌─────────┐      ┌─────────┐      ┌─────────┐
│  お客   │ ───→ │ウェイター│ ───→ │  厨房   │
│（アプリ）│ ←─── │ （API） │ ←─── │（DB等） │
└─────────┘      └─────────┘      └─────────┘
   注文            伝達・運搬         調理
```

- **メニュー** → 「何を注文できるか」のリスト
- **注文方法** → 「どうやって頼むか」のルール
- **ウェイター** → 注文を厨房に伝え、料理を運ぶ人（= API）
- **料理** → 欲しかったもの（= データ）

厨房の中で何が起きているか、お客は知らなくていい。「カルボナーラください」と言えば、カルボナーラが出てくる。それが API です。

## スマホアプリの裏側で何が起きているか

天気予報アプリを開いたとき、何が起きているか。

アプリは「東京の天気を教えて」と**どこかに聞いている**。その「どこか」は気象庁のサーバーかもしれないし、別の天気サービスかもしれない。

でも、アプリ自身が天気を観測しているわけではありません。

```
[あなたのスマホ]  →  「東京の天気は？」  →  [天気サービスのAPI]
                 ←  「晴れ、最高気温18度」 ←
```

これが API の正体です。

アプリは「聞き方」さえ知っていれば、世界中のサービスから情報をもらえる。自分で気象観測所を建てる必要はありません。

## 「APIを作る」とは何をしているのか

会議で「まずAPIを作る」と言われたとき、エンジニアは何をしようとしているのか。

レストランの例に戻ります。

新しいレストランをオープンするとき、最初に決めることは何でしょう？

1. **メニューを決める**：何を提供するか
2. **注文方法を決める**：どうやって頼むか（口頭？タブレット？）
3. **厨房を準備する**：実際に料理を作れるようにする

「APIを作る」とは、この1と2を決めることです。

- 「ユーザー情報を取得する」という注文ができるようにする
- 「新しい問い合わせを登録する」という注文ができるようにする
- 「過去の履歴を検索する」という注文ができるようにする

メニュー（何ができるか）と注文方法（どう頼むか）が決まれば、厨房（バックエンド）と客席（フロントエンド）を同時に作り始められます。

だから「まずAPIから作る」んです。

## なぜエンジニアはAPIを大事にするのか

### 1. 分業できるようになる

APIが決まれば、画面を作る人とサーバーを作る人が、お互いの完成を待たずに作業できます。

「カルボナーラを注文したら、カルボナーラが来る」というルールが決まっていれば、厨房担当は「カルボナーラの作り方」を、ホール担当は「お客様への対応」を、それぞれ並行して準備できます。

### 2. 変更に強くなる

厨房を最新の機材に入れ替えても、お客の注文方法は変わりません。

同じように、APIの「外側」が同じなら、「内側」をどれだけ改善しても、使っている側に影響がありません。

### 3. 安全に情報をやり取りできる

お客が直接厨房に入るのは危険です。包丁もあれば火もある。

APIは「窓口」です。直接データベースを触らせず、決められた方法でだけ情報をやり取りする。だから安全なんです。

## 明日から使える知識

会議でこんな発言が出たら、こう理解してください。

| 発言 | 意味 |
|:---|:---|
| 「APIを叩く」 | 注文を出す（リクエストを送る） |
| 「APIを生やす」 | 新しいメニューを追加する |
| 「APIの仕様を決める」 | メニューと注文方法を決める |
| 「APIが返してくる」 | 注文した料理が届く（データが返ってくる） |
| 「APIがエラーを返す」 | 「その注文は受けられません」と言われる |

---

## まとめ

APIとは、ソフトウェア同士が会話するための「注文の仕組み」です。

- **メニュー**があり（何ができるか）
- **注文方法**があり（どう頼むか）
- **料理が届く**（データが返ってくる）

レストランのウェイターのように、厨房の複雑さを隠して、お客には「注文すれば届く」というシンプルな体験を提供する。それがAPIの本質です。

次に会議で「API」という言葉が出てきたら、「注文の仕組みを作ってるんだな」と思ってください。きっと話についていけるはずです。
